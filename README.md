# CS305
This repository contains my completed Artemis Financial Vulnerability Assessment Report. In this project, I performed a security assessment for Artemis Financial, a company that provides financial services such as savings plans, retirement planning, investments, and insurance. Because the company handles sensitive customer information, it wanted to identify security vulnerabilities within its software and improve the overall security of its application. I analyzed the application's security requirements, performed a manual code review, identified potential threats, completed a dependency check, and created a mitigation plan to address the risks that were discovered.

One of the things I did well during this project was identifying several security vulnerabilities through manual code inspection. I found issues such as hard-coded database credentials, missing input validation, possible SQL injection risks, missing authentication on REST endpoints, weak encapsulation, and outdated third-party libraries. Secure coding is important because it helps protect sensitive customer information, reduces the risk of cyberattacks, and helps organizations maintain customer trust while meeting security and compliance requirements. The most helpful part of the project was learning how to analyze code from a security perspective and understanding how dependency scanning can identify vulnerabilities in external libraries that may not be obvious during manual review.

To improve the application's security, I recommended removing hard-coded credentials, validating user input, using parameterized SQL queries, improving error handling, enforcing HTTPS and TLS, adding authentication and authorization, strengthening encapsulation, and updating vulnerable dependencies. In future projects, I would continue using manual code reviews, vulnerability assessments, static analysis tools such as OWASP Dependency-Check, and secure coding best practices to identify risks and choose appropriate mitigation strategies. These tools provide multiple layers of security by identifying weaknesses in both the application code and the third-party libraries it depends on.

Although Project One focused primarily on assessing vulnerabilities instead of modifying the application, I verified my findings by performing both a manual code review and a dependency scan to confirm known security issues within the project's dependencies. These assessment techniques helped ensure my recommendations were based on documented vulnerabilities rather than assumptions. Throughout this project, I used Spring Boot, Maven, OWASP Dependency-Check, Eclipse, secure coding principles, and industry best practices for vulnerability analysis. These are all resources and practices that will continue to be valuable in future software development and cybersecurity projects.

This project is a good example to share with future employers because it demonstrates my ability to evaluate software security, identify vulnerabilities, assess potential risks, recommend practical mitigation strategies, and document my findings in a professional security assessment report. It also shows my understanding of secure software development and the importance of protecting sensitive information throughout the software development lifecycle.
